# Requirements

## 기능 요구사항

### 데이터베이스 모델

- H2 Database와 JPA를 사용해 상품과 주문 도메인의 데이터베이스 모델을 구현한다.
- 상품은 `products` 테이블로 저장하며 상품명, 가격, 재고수량, 카테고리, 생성일시, 수정일시를 가진다.
- 주문은 `orders` 테이블로 저장하며 주문상태, 생성일시, 수정일시를 가진다.
- 주문 항목은 `order_items` 테이블로 저장하며 주문, 상품, 주문수량, 주문 당시 가격을 가진다.
- `Order`와 `Product`는 직접 다대다 관계로 연결하지 않고 `OrderItem`을 통해 연결한다.
- 주문 당시 가격은 `OrderItem.orderPrice`에 저장해 상품 가격 변경 이후에도 주문 내역의 의미가 유지되어야 한다.
- `Product`, `Order`, `OrderItem`은 JPA entity로 구현하되 API 응답으로 직접 반환하지 않는다.
- `Category`와 `OrderStatus`는 enum으로 구현하고 DB에는 문자열로 저장한다.
- 상품 카테고리는 초기 지원 enum 값으로 `FOOD`, `FASHION`, `ELECTRONICS`, `BOOK`, `ETC`를 사용한다.
- 생성일시와 수정일시는 entity 공통 감사 필드로 관리한다.

### 상품

- 상품은 상품명, 가격, 재고수량, 카테고리를 가진다.
- 상품을 등록할 수 있다.
- 상품을 수정할 수 있다.
- 상품을 단건 조회할 수 있다.
- 상품 목록을 조회할 수 있다.
- 특정 카테고리의 상품 목록을 조회할 수 있다.
- 상품 목록 조회는 페이징을 지원한다.
- 재고 수량이 부족한 경우 주문 완료는 실패해야 한다.

### 주문

- 주문은 하나 이상의 상품을 포함할 수 있다.
- 주문 생성 시 기본 상태는 대기다.
- 주문 상태는 대기, 접수, 완료, 취소 중 하나다.
- 주문 상태는 생성 이후 변경할 수 있다.
- 비즈니스적으로 올바르지 않은 상태 변경은 허용하지 않는다.
- 현재 상태와 동일한 상태로 변경하는 요청은 허용 상태 전이에 포함하지 않는다.
- 동일 상태 변경 요청은 멱등 성공으로 처리하지 않고 허용되지 않은 주문 상태 변경 예외로 응답한다.
- 주문이 완료되면 주문에 포함된 상품들의 재고가 차감된다.
- 주문이 취소될 경우 이미 차감된 재고가 있다면 복구된다.
- 주문을 단건 조회할 수 있다.
- 주문 목록을 조회할 수 있다.
- 주문 상태별로 조회할 수 있다.
- 주문 기간별로 조회할 수 있다.
- 주문 목록 조회는 페이징을 지원한다.

## API 요구사항

### 상품 API

- `POST /api/products`: 상품 등록
- `PUT /api/products/{productId}`: 상품 수정
- `GET /api/products/{productId}`: 상품 단건 조회
- `GET /api/products`: 상품 목록 조회
- `GET /api/products?category={category}`: 카테고리 기준 상품 조회

### 주문 API

- `POST /api/orders`: 주문 생성
- `PATCH /api/orders/{orderId}/status`: 주문 상태 변경
- `GET /api/orders/{orderId}`: 주문 단건 조회
- `GET /api/orders`: 주문 목록 조회
- `GET /api/orders?status={status}`: 주문 상태별 조회
- `GET /api/orders?from={from}&to={to}`: 기간별 주문 조회
- 주문 목록 조회는 `status`, `from`, `to`, `page`, `size` 조건을 함께 사용할 수 있다.
- `from`과 `to`는 ISO local date-time 형식으로 받으며, `createdAt >= from`, `createdAt <= to` 기준으로 포함 조회한다.

## 데이터 무결성 요구사항

- 상품 가격은 0 이상이어야 한다.
- 상품 재고수량은 0 이상이어야 한다.
- 주문 항목 수량은 1 이상이어야 한다.
- 주문은 하나 이상의 주문 항목을 가져야 한다.
- 주문 항목은 존재하는 상품만 참조할 수 있다.
- 존재하지 않는 상품에 대한 주문 요청은 실패해야 한다.
- 재고 수량이 부족한 상품 주문 완료는 실패해야 한다.
- 허용되지 않은 주문 상태 변경 요청은 실패해야 한다.
- 주문 상태 변경과 재고 변경은 하나의 트랜잭션에서 처리해야 한다.
- 잘못된 요청은 일관된 에러 응답으로 반환해야 한다.

## 동시성 요구사항

- 여러 주문이 동시에 발생해도 상품 재고가 정확하게 차감되어야 한다.
- 재고가 음수가 되면 안 된다.
- 동시성 해결 방식은 코드와 README에 설명되어야 한다.
- 같은 주문에 상태 변경 요청이 동시에 들어와도 상태 변경과 재고 차감 또는 복구가 중복 적용되면 안 된다.
- 주문 상태 변경 API는 대상 주문을 비관적 쓰기 락으로 조회한 뒤 상태 전이 가능 여부를 검증한다.
- 재고 변경이 필요한 상태 전이에서는 주문 row 락을 획득한 뒤, 주문 항목의 상품 ID를 정렬해 상품 row 비관적 락을 획득한다.
- 처리 순서는 `주문 row lock -> 상품 row lock -> 상태 전이 검증 -> 재고 검증/변경 -> 주문 상태 변경`으로 한다.

## 테스트 요구사항

- 주요 비즈니스 로직에 대해 테스트 코드를 작성한다.
- 테스트 대상과 의도가 코드 또는 설명으로 드러나야 한다.
- 주문 상태 전이, 재고 차감, 재고 부족, 존재하지 않는 상품, 동시성 처리는 반드시 테스트한다.

## 성능 요구사항

- 상품 목록과 주문 목록은 페이징을 적용한다.
- 주문 데이터가 많아질 경우 상태별 조회, 기간별 조회의 성능 문제와 개선 방안을 README에 설명한다.
- 주문 상세 조회 또는 목록 조회에서 N+1 문제가 발생하지 않도록 고려한다.
- 주문 상태별 조회를 위해 `orders.status` 인덱스를 고려한다.
- 주문 기간별 조회와 기본 정렬을 위해 `orders.created_at` 인덱스를 고려한다.
- 주문 상세 조회의 주문 항목 접근을 위해 `order_items.order_id` 인덱스를 고려한다.
- 주문 상세 조회는 fetch join, EntityGraph, DTO projection 중 하나로 N+1을 방지한다.

## 데이터베이스 테이블 요구사항

### `products`

- `id`: 상품 식별자, PK
- `name`: 상품명, 필수, 최대 100자
- `price`: 상품 가격, 필수, 0 이상
- `stock_quantity`: 재고수량, 필수, 0 이상
- `category`: 상품 카테고리, 필수
- `created_at`: 생성일시, 필수
- `updated_at`: 수정일시, 필수

### `orders`

- `id`: 주문 식별자, PK
- `status`: 주문 상태, 필수
- `created_at`: 생성일시, 필수
- `updated_at`: 수정일시, 필수

### `order_items`

- `id`: 주문 항목 식별자, PK
- `order_id`: 주문 FK, 필수
- `product_id`: 상품 FK, 필수
- `quantity`: 주문수량, 필수, 1 이상
- `order_price`: 주문 당시 상품 가격, 필수, 0 이상

## 결정된 정책

- 주문 생성 시에는 재고를 차감하지 않는다.
- `COMPLETED` 상태로 변경될 때 주문 항목 수량만큼 재고를 차감한다.
- `COMPLETED` 상태의 주문이 `CANCELED` 상태로 변경될 때 차감된 재고를 복구한다.
- `PENDING` 또는 `ACCEPTED` 상태에서 취소될 경우 재고 복구를 수행하지 않는다.
- 허용 상태 전이는 `PENDING -> ACCEPTED`, `ACCEPTED -> COMPLETED`, `PENDING -> CANCELED`, `ACCEPTED -> CANCELED`, `COMPLETED -> CANCELED`로 제한한다.
- 동일 상태 변경 요청은 허용하지 않는다.
- 주문 상태 변경 시 대상 주문 row에 비관적 락을 적용한다.
- 주문 완료 처리 시 재고 차감 대상 상품 row에 비관적 락을 적용한다.
- 완료 주문 취소처럼 재고 복구가 필요한 상태 전이에서도 상품 row에 비관적 락을 적용한다.
- 여러 상품을 동시에 차감할 때는 상품 ID 기준으로 정렬해 락 획득 순서를 일정하게 유지한다.
