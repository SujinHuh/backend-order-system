# Requirements

## 기능 요구사항

### 1. 주문 row 락

- 주문 상태 변경 유스케이스는 대상 주문을 비관적 쓰기 락으로 조회해야 한다.
- 락이 적용된 주문 조회는 `OrderRepository` 책임으로 둔다.
- 존재하지 않는 주문은 기존과 동일하게 `EntityNotFoundException`으로 처리한다.

### 2. 상품 row 락

- 재고 변경이 필요한 상태 전이에서는 주문 항목에 포함된 상품 row를 비관적 쓰기 락으로 조회해야 한다.
- 재고 차감이 필요한 상태 전이:
  - `ACCEPTED -> COMPLETED`
- 재고 복구가 필요한 상태 전이:
  - `COMPLETED -> CANCELED`
- 재고 변경이 필요 없는 상태 전이에서는 상품 row 락을 획득하지 않는다.
- 여러 상품이 포함된 주문은 상품 ID 오름차순으로 락을 획득해야 한다.
- 상품 락 조회는 `ProductRepository` 책임으로 둔다.

### 3. 트랜잭션 처리 순서

- 주문 상태 변경은 하나의 service 트랜잭션 안에서 처리한다.
- 처리 순서는 다음을 따른다.
  1. 주문 row 비관적 쓰기 락 획득
  2. 현재 상태와 대상 상태 기준으로 재고 변경 필요 여부 판단
  3. 재고 변경이 필요하면 주문 항목의 상품 ID를 정렬해 상품 row 비관적 쓰기 락 획득
  4. 상태 전이 가능 여부 검증
  5. 재고 부족 여부 검증 및 재고 차감/복구
  6. 주문 상태 변경
- 위 단계는 같은 트랜잭션에서 완료되어야 한다.

### 4. 기존 상태 전이/재고 정책 유지

- 주문 생성 시에는 재고를 차감하지 않는다.
- `COMPLETED`로 변경될 때만 주문 항목 수량만큼 재고를 차감한다.
- `COMPLETED -> CANCELED`일 때만 차감된 재고를 복구한다.
- 동일 상태 변경 요청은 성공으로 처리하지 않는다.
- 허용되지 않은 상태 전이는 기존처럼 400 계열 비즈니스 예외로 처리한다.

## 비기능 요구사항 또는 품질 요구사항

- 동시성 정합성이 명시적 테스트로 검증되어야 한다.
- 동시 테스트는 외부 DB나 Testcontainers 없이 프로젝트 기본 H2 테스트 환경에서 수행한다.
- 재고가 음수가 되거나 같은 주문에서 재고가 중복 차감되는 결과가 없어야 한다.
- 불필요한 락 범위를 넓히지 않기 위해 재고 변경이 없는 상태 전이에서는 상품 row 락을 생략한다.
- 여러 상품 락 획득 순서는 deadlock 가능성을 줄이기 위해 항상 상품 ID 오름차순으로 유지한다.

## 입력/출력

- 입력 API:
  - `PATCH /api/orders/{id}/status`
  - Request body: `{ "status": "COMPLETED" }` 또는 다른 `OrderStatus`
- 출력 API:
  - 기존 `OrderResponse` 형식을 유지한다.
- Repository 입력:
  - 주문 ID
  - 상품 ID 목록

## 제약사항

- `DEC-001`과 `DEC-002`의 정책을 변경하지 않는다.
- 동시성 전략은 비관적 쓰기 락으로 구현한다.
- 주문 row 락을 상품 row 락보다 먼저 획득한다.
- 상품 row 락은 상품 ID 정렬 순서로 획득한다.
- 이번 태스크에서 QueryDSL 조건 조회, 주문 조회 API, 최종 README 정리는 구현하지 않는다.
- 기존 API URL과 응답 DTO는 변경하지 않는다.

## 예외 상황

- 주문이 존재하지 않으면 `EntityNotFoundException`이 발생한다.
- 상품 row 락 조회 중 주문 항목 상품이 존재하지 않으면 `EntityNotFoundException` 또는 기존 not found 예외 정책에 맞는 예외가 발생해야 한다.
- 재고 부족 시 `NotEnoughStockException`이 발생하고 트랜잭션은 롤백되어야 한다.
- 허용되지 않은 상태 전이는 기존 `BusinessException` 흐름을 유지한다.
- 동시 요청 중 일부가 재고 부족 또는 상태 전이 오류로 실패하더라도 최종 재고와 주문 상태는 일관되어야 한다.

## 성공 기준

- `OrderService.updateStatus`가 주문 row 비관적 쓰기 락 기반으로 동작한다.
- 재고 변경이 필요한 상태 전이에서 상품 row 비관적 쓰기 락이 적용된다.
- 같은 상품에 대한 동시 주문 완료 요청에서 성공한 주문 수만큼만 재고가 차감된다.
- 같은 상품에 대한 동시 주문 완료 요청 이후 최종 재고는 음수가 아니다.
- 같은 주문에 대한 동시 완료 요청에서 재고가 중복 차감되지 않는다.
- 기존 주문 상태 변경/재고 차감/복구 테스트가 계속 통과한다.
- `./mvnw test`가 통과한다.

## Decision 반영 후보

- 현재 태스크는 기존 `DEC-002` 정책을 구현하는 작업이다.
- 동시성 전략 자체를 변경하지 않으므로 새 decision 생성은 필요하지 않다.
- Phase 5에서 구현 결과가 `DEC-002`, `architecture.md`, `testing_profile.md`, README와 불일치하는지 확인하고 필요한 문서 현행화를 판단한다.
