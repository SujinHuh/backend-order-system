# Issue

## 배경

`feat/order-status-stock`까지 주문 상태 변경과 재고 차감/복구 흐름은 구현되었다. 현재 `OrderService.updateStatus`는 일반 `findById`로 주문을 조회한 뒤 상태 전이와 재고 변경을 같은 트랜잭션에서 수행한다.

그러나 여러 요청이 같은 주문 또는 같은 상품 재고를 동시에 변경하면 아래 위험이 남는다.

- 같은 주문에 완료 요청이 동시에 들어올 때 재고가 중복 차감될 수 있다.
- 여러 주문이 같은 상품 재고를 동시에 차감할 때 재고 부족 판단과 차감이 직렬화되지 않을 수 있다.
- 완료 주문 취소로 재고 복구가 발생하는 흐름도 상품 row 변경을 동기화해야 한다.

프로젝트 결정 문서 `DEC-002`는 주문 상태 변경 시 주문 row 비관적 쓰기 락을 먼저 획득하고, 재고 변경이 필요한 상태 전이에서는 상품 row 비관적 쓰기 락을 상품 ID 정렬 순서로 획득하도록 정의한다.

## 요청사항

- `feat/order-concurrency-control` 브랜치에서 주문 상태 변경/재고 변경 동시성 제어를 구현한다.
- 주문 상태 변경 API에서 대상 주문 row를 `PESSIMISTIC_WRITE`로 조회한다.
- `COMPLETED` 전환과 `COMPLETED -> CANCELED` 전환처럼 재고 변경이 필요한 경우 대상 상품 row를 `PESSIMISTIC_WRITE`로 조회한다.
- 여러 상품을 다루는 주문은 상품 ID 오름차순 기준으로 상품 row 락을 획득한다.
- 락 획득, 상태 전이 검증, 재고 검증/차감/복구, 주문 상태 변경은 하나의 트랜잭션 안에서 처리한다.
- 같은 상품에 대한 동시 주문 완료 요청에서도 재고가 음수가 되지 않음을 테스트한다.
- 같은 주문에 대한 동시 완료 요청에서 재고 중복 차감이 발생하지 않음을 테스트한다.

## 비범위

- 주문 조회 API 및 QueryDSL 조건 조회 구현은 `feat/query-apis` 범위로 남긴다.
- 상품 카테고리 조건 조회 구현은 `feat/query-apis` 범위로 남긴다.
- 최종 README 제출 문서 정리는 `docs/final-readme` 범위로 남긴다.
- 동시성 전략을 낙관적 락 또는 조건부 update 방식으로 변경하지 않는다.
- 주문 생성 시점 재고 예약 또는 재고 선점 정책은 도입하지 않는다.
- 전역 예외 코드 체계 세분화는 이번 태스크에서 확장하지 않는다.

## 승인 또는 제약 조건

- `DEC-001`의 주문 상태 전이와 재고 처리 정책을 변경하지 않는다.
- `DEC-002`의 비관적 락 정책을 따른다.
- 트랜잭션 경계는 service public 메서드에 둔다.
- Controller에서 repository를 직접 호출하지 않는다.
- Entity를 API 응답으로 직접 반환하지 않는다.
- `./mvnw test`가 통과해야 한다.
