# Lightweight Task Policy

## 목적

이 문서는 작은 태스크에 대해 경량 운영 예외를 둘지 판단하는 기준과 허용 범위를 정의한다.

## 언제 다시 읽는가

- 작은 태스크로 보일 수 있는 변경이지만 예외 적용 여부가 불분명할 때만 참조한다.
- 기본값은 전체 Harness Phase를 그대로 따르는 것이며, 이 문서는 상시 재참조 문서가 아니다.

## 적용 대상 후보

아래 조건을 모두 만족하면 작은 태스크 후보로 본다.

- 변경 범위가 국소적이다.
- 새로운 아키텍처 계약, 외부 인터페이스, 데이터 스키마 변경이 없다.
- 다수 레이어를 가로지르는 구현이 아니다.
- 회귀 위험이 낮고 영향 범위가 명확하다.
- 요구사항 해석에 큰 불확실성이 없다.

## 경량 판정 체크리스트

아래 질문에 모두 `YES`일 때만 경량 운영 예외 후보로 진행할 수 있다.

- 외부 계약(API, 이벤트, CLI, 파일 포맷, 공개 인터페이스) 변경이 없는가
- 데이터 스키마, 저장 구조, 마이그레이션 변경이 없는가
- 구현이 다수 레이어를 동시에 가로지르지 않는가
- 회귀 위험이 낮고 영향 범위를 빠르게 설명할 수 있는가
- 요구사항 해석, 승인 범위, 성공 기준에 큰 불확실성이 없는가

하나라도 `NO`이면 경량 운영 예외를 적용하지 않고 표준 Phase 흐름으로 진행한다.

## 기본 원칙

- 작은 태스크라도 기본값은 전체 Harness Phase를 그대로 따르는 것이다.
- 경량 운영 예외는 사용자 승인 없이는 적용하지 않는다.
- 경량 운영 예외를 적용해도 테스트, 감사, 검증 책임 자체를 생략한 것으로 간주하지 않는다.
- 경량 운영 예외 적용 여부와 사유는 `implementation_notes.md`에 기록한다.
- 경량 운영 중에도 입력 문서나 핵심 산출물이 바뀌면 먼저 가장 이른 영향 Phase, stale 처리할 감사/승인, 잠글 산출물을 다시 선언한다.
- 경량 운영 예외 승인도 최신 입력 근거가 바뀌면 stale 이 될 수 있으며, stale 상태가 해소되기 전에는 다음 Phase 문서나 close-out 문서를 먼저 수정하지 않는다.

## 경량 수행 최소 절차

경량 운영 예외가 승인되더라도 아래 절차는 항상 유지한다.

- 관련 테스트는 실제로 수행하고 결과를 남긴다.
- 변경 부분 중심 감사는 필수로 수행한다.
- 전체 흐름 영향 요약을 남긴다.
- 각 작업 단위가 끝날 때마다 `implementation_notes.md`의 `진행 로그`를 즉시 갱신한다.
- 경량 검토 결과는 같은 작업에서 `implementation_notes.md`의 `경량 검토 기록` 섹션에 즉시 남긴다.
- self-healing 이 발동되면 `implementation_notes.md`에 변경 원인, 가장 이른 영향 Phase, stale 처리 범위, 잠긴 문서 범위를 함께 남긴다.

## 허용 가능한 예외 범위

사용자가 명시적으로 승인한 경우에만 아래 범위를 검토할 수 있다.

- Phase 1 산출물을 간략화하되 `requirements.md`와 `plan.md`의 핵심 판단 근거는 남긴다.
- Phase 2에서 실제 필요한 레이어만 더 좁게 선택해 진행한다.
- Phase 5 문서화 범위를 변경 파일과 작업 로그 중심으로 축소한다.

## 경량 불가 또는 즉시 Full 전환 트리거

아래 중 하나라도 발견되면 경량 운영을 중단하고 표준 Phase 흐름으로 전환한다.

- API, 이벤트, CLI, 파일 포맷, 공개 인터페이스 계약 변경이 필요해짐
- 데이터 스키마, 저장 구조, 마이그레이션 변경이 필요해짐
- 보안, 성능, 운영 안정성에 임계 영향 가능성이 드러남
- 구현 범위가 다중 레이어 또는 다중 모듈로 확장됨
- 승인된 범위를 넘는 추가 요구사항이나 범위 확장이 발견됨
- 요구사항, 승인 조건, 성공 기준 해석이 불확실해짐
- 경량 승인 이후 입력 문서나 핵심 산출물이 바뀌어 경량 판정 근거가 stale 됨
- Phase 5 close-out 또는 다음 Phase 문서를 먼저 수정해야만 진행되는 상태가 됨

전환이 발생하면 전환 사유와 이후 적용할 표준 Phase 계획을 즉시 기록한다.

- 경량 승인 또는 변경 부분 중심 감사가 stale 되면, 기본값은 표준 Phase 흐름으로 복귀하는 것이다.

## 허용하지 않는 예외

- 사용자 승인 없는 Phase 생략
- 사용자 승인 없는 감사 생략
- 테스트 필요 변경에서 테스트 생략
- 검증 결과와 남은 리스크 기록 생략
- 구조 변경이나 계약 변경을 경량 태스크로 오인해 축소 처리하는 것

## 경량 검토 기록 템플릿

경량 운영 예외를 검토했다면 `implementation_notes.md`의 `경량 검토 기록` 섹션에 아래 형식으로 남긴다.

- 작은 태스크로 본 근거
- 경량 적용 승인 여부
- 실제 축소한 범위
- 유지한 테스트
- 유지한 감사
- 전체 흐름 영향 요약
- 남은 리스크
- Full 전환 조건 또는 승격 조건
