# Coding Guidelines Policy

## 목적

여러 프로젝트에서 공통으로 재사용할 수 있는 코드 작성 기준을 common process policy로 정의한다.

## 기본 원칙

- 코드는 읽기 쉬워야 한다.
- 함수와 모듈은 하나의 주된 책임에 집중해야 한다.
- 중복은 줄이되, 과도한 추상화로 가독성을 해치지 않는다.
- 구현은 현재 요구사항을 충족하는 최소 복잡도로 작성한다.
- 승인되지 않은 선제 일반화는 하지 않는다.

## 코드 작성 기준

- 이름은 역할이 드러나야 한다.
- 숨은 부수효과를 피한다.
- 반환값과 실패 조건은 예측 가능해야 한다.
- 의미 없는 유틸 분리는 지양한다.
- 불필요한 레이어 추가를 피한다.

## 구현 품질 기준

- 함수와 클래스는 단일한 책임이 드러나야 하며, 하나의 변경 이유만 갖도록 구성한다.
- 한 모듈이나 클래스에 여러 레이어 책임을 섞지 않는다.
- 새로운 추상화는 현재 TASK에서 반복 또는 경계 명확화에 실질적으로 필요할 때만 도입한다.
- 구현은 기존 레이어 책임과 의존성 방향을 흐리지 않아야 한다.
- 실패 경로와 예외 케이스는 happy path와 동일한 수준으로 의도를 드러내야 한다.
- 테스트는 구현 세부사항에 과도하게 결합되지 않도록 작성한다.

## 리팩터링 범위 제한 규칙

- 별도 리팩터링 태스크가 아니면 현재 TASK 수행에 필요한 부분만 리팩터링한다.
- 현재 수정 대상과 직접 관련 없는 기존 중복 코드는 이번 TASK에서 정리하지 않는다.
- 현재 수정 대상과 직접 관련 없는 네이밍 정리, 포맷 정리, 파일 이동, 추상화 도입은 하지 않는다.
- 기능 구현 또는 버그 수정에 반드시 필요한 정리라면 그 영향 범위를 최소화한다.
- 필요 최소 범위를 넘는 구조 개선이 필요하면 `implementation_notes.md`에 후속 태스크 후보로 남긴다.

허용 가능한 최소 정리 기준:

- 현재 TASK 요구사항을 구현하거나 버그를 수정하지 않으면 변경 자체가 성립하지 않는 정리
- 현재 수정 파일 내부에서 테스트 가능성이나 기존 아키텍처 규칙 준수를 위해 불가피한 정리
- 현재 변경 코드가 의존하는 바로 인접한 중복 제거 또는 함수 추출처럼 영향 범위가 국소적인 정리

현재 TASK 범위 밖 리팩터링 기준:

- 현재 수정 대상과 직접 관련 없는 기존 중복 제거
- 전체 네이밍 정리, 광범위한 파일 이동, 공통 유틸 추출, 미래 확장을 위한 선제 추상화
- 현재 TASK를 수행하지 않아도 독립적으로 진행 가능한 구조 개선

## 에러 처리 규칙

- 실패는 레이어 책임에 맞는 형태로 표현한다.
- 외부 SDK, HTTP 클라이언트, DB 클라이언트의 원시 에러를 core layer에 그대로 노출하지 않는다.
- 경계에서는 외부 실패를 현재 유스케이스가 이해할 수 있는 의미 있는 실패로 변환한다.
- 실패 메시지는 원인과 책임 경계를 드러내야 하지만 민감정보를 직접 포함하지 않는다.
- user-facing 응답 메시지와 내부 원인 표현은 분리할 수 있어야 한다.
- 정상 흐름 제어를 위해 예외를 남용하지 않는다.
- 실패를 잡고 무시하거나 의미 없는 일반 에러로 덮어쓰지 않는다.

## 로깅 및 관측성 규칙

- 로그는 디버깅, 장애 대응, 추적에 필요한 목적이 있을 때만 남긴다.
- 같은 실패를 여러 레이어에서 중복 기록하지 않는다.
- 로그에는 요청 식별자, 작업 식별자, 유스케이스 식별 정보처럼 추적에 필요한 최소 상관관계 정보를 포함한다.
- 토큰, 비밀번호, 세션 값, 원문 payload, 민감한 개인식별정보는 로그에 직접 남기지 않는다.
- 정상 흐름의 모든 단계에 과도한 로그를 남기지 않는다.
- user-facing 응답 메시지와 내부 로그 메시지는 분리할 수 있어야 한다.

## Boundary Translation Rules

- 경계가 존재하는 곳에서는 경계 바로 앞 레이어가 번역 책임을 가진다.
- 외부 모델, 저장 포맷, transport 포맷을 core가 직접 사용하게 두지 않는다.
- “구조가 비슷하다”는 이유만으로 다른 레이어의 DTO나 외부 모델을 재사용하지 않는다.
- 번역 과정에서는 현재 유스케이스에 필요한 정보만 유지하고 외부 포맷 세부사항은 제거한다.
- 번역 책임을 하위 레이어나 호출자에게 떠넘겨 여러 레이어가 같은 외부 포맷을 직접 이해하게 만들지 않는다.

## 테스트 기준

- 테스트 공통 기준은 `docs/process/common/testing_policy.md`를 따른다.
- 테스트 산출물과 위임 책임 기록 규칙은 `docs/process/common/artifact_policy.md`를 따른다.
- 실제 테스트 명령, coverage 목표, 환경 의존성은 project overlay에서 정의한다.

## 금지 사항

- 승인되지 않은 범위 확장
- 쓰이지 않는 추상화 추가
- 근거 없는 premature optimization
- 책임이 섞인 구현
- 테스트 없는 핵심 로직 변경

## Review Checklist

구현 후 최소한 아래 질문을 스스로 점검한다.

- 현재 변경이 승인된 TASK 범위를 벗어나지 않았는가
- 현재 TASK와 직접 관련 없는 기존 중복 제거, 구조 정리, 선제 추상화를 함께 넣지 않았는가
- 레이어 책임이 섞이지 않았는가
- 외부 모델과 내부 모델의 번역 경계가 흐려지지 않았는가
- 테스트가 구현 세부사항이 아니라 동작과 계약을 검증하는가
- 실패 경로와 예외 케이스가 happy path와 같은 수준으로 점검되었는가
- 에러 처리에서 원시 외부 에러 유입, 예외 남용, 민감정보 노출이 없는가
- 로그가 민감정보를 노출하지 않고, 상관관계 정보와 목적에 맞게 남겨졌는가
- 불필요한 복잡도, 과도한 추상화, 책임 과다가 없는가
- 새 계약, 구조 결정, 범위 밖 개선 사항이 있으면 문서 반영 또는 후속 태스크 후보 기록이 되었는가
