# Design Quality Policy

## 목적

이 문서는 Phase 2 구현과 감사에서 코드가 현재 요구사항을 충족하는 것뿐 아니라, 읽기 쉬운 구조와 변경 가능한 설계를 유지하는지 판단하는 공통 기준을 정의한다.

## 언제 다시 읽는가

- Phase 2 구현 시작 전에 다시 확인한다.
- 책임 분리, 추상화 수준, 중복 처리, 구조 개선 범위가 애매할 때 다시 읽는다.
- Phase 2 감사 직전에 설계 품질 기준을 다시 확인할 때 참조한다.

## 기본 원칙

- 코드는 현재 요구사항을 만족하는 최소 복잡도로 유지한다.
- 설계 품질은 "예쁘게 보이는 구조"가 아니라, 읽기 쉬움, 변경 용이성, 책임 분리, 테스트 가능성을 함께 만족하는 상태다.
- 현재 TASK와 직접 관련 없는 대규모 재설계나 선제 추상화는 design quality 명분으로 함께 넣지 않는다.

## 이름과 표현 규칙

- 이름은 의도와 역할을 드러내야 한다.
- 같은 개념에는 같은 단어를 사용하고, 비슷해 보이지만 다른 의미를 같은 이름으로 섞지 않는다.
- `Manager`, `Helper`, `Common`, `Util`처럼 책임이 넓고 모호한 이름은 지양한다.
- 함수/변수/타입 이름만으로 의도를 충분히 드러낼 수 있으면 설명용 주석보다 코드 표현을 우선한다.

## 이름과 경계 판정 질문

- 이름이 현재 타입/함수의 역할을 한 문장으로 설명할 수 있는가
- 같은 타입/함수 이름에 validation, orchestration, persistence, mapping 같은 서로 다른 책임이 함께 암시되는가
- 외부 DTO/response/entity를 해석하는 책임과 domain 판단이 같은 함수/클래스에 같이 들어가 있는가
- 호출자가 두 단계 이상 필드/객체를 따라가며 foreign data를 직접 꺼내야 하는가
- 위 질문 중 둘 이상에 `예`면 이름 정리, 책임 이동, 경계 캡슐화를 우선 검토한다

## 함수와 메서드 설계 규칙

- 함수는 가능한 한 한 가지 일만 하게 유지한다.
- 한 함수 안의 추상화 수준은 가능한 한 한 단계로 맞춘다.
- 긴 함수, 과도한 중첩, 긴 매개변수 목록은 더 작은 책임 단위로 분리할 수 있는지 먼저 검토한다.
- 플래그 인수, 출력 인수, 숨은 부수효과는 지양한다.
- 명령과 조회를 섞지 않고, 가능하면 상태 변경과 값 조회를 분리한다.

## 함수 분리 판정 질문

- 현재 함수가 하나의 동사/행위로 자연스럽게 요약되는가
- 입력 검증, 계산, 외부 호출, 저장, 응답 포맷팅처럼 서로 다른 책임이 함께 들어가 있는가
- 현재 변경으로 같은 함수 안에 독립 변경 이유가 두 가지 이상 생겼는가
- 불리언 플래그나 분기 추가 때문에 사실상 다른 두 작업을 한 함수가 수행하게 되었는가
- 위 질문 중 둘 이상에 `예`면 함수 분리 또는 책임 이동을 우선 검토한다

## 책임 분리와 응집도 규칙

- 함수, 클래스, 모듈은 하나의 주된 변경 이유를 갖도록 유지한다.
- 한 모듈 안에 domain/application/infrastructure 책임을 섞지 않는다.
- 경계 번역, orchestration, domain 규칙, 외부 연동 코드를 한 함수나 클래스에 함께 몰아넣지 않는다.
- 같이 바뀌는 코드는 가깝게 두고, unrelated concern은 분리한다.

## 추상화와 재사용 규칙

- 반복이 실제로 존재하고 현재 TASK에서 이득이 있을 때만 추상화를 도입한다.
- 재사용 근거 없는 interface, base class, wrapper, factory, helper 추가는 지양한다.
- 추측성 일반화, 미래 확장 대비용 추상화, 아직 하나뿐인 구현체를 위한 과한 계층은 기본 금지다.
- 중복을 제거할 때는 "삭제", "국소 통합", "그대로 유지" 중 어느 선택이 현재 TASK에서 가장 단순한지 먼저 판단한다.

## 추상화 도입 판정 질문

- 현재 TASK 안에서 같은 개념의 중복이 두 번 이상 실제로 나타나는가
- 새 추상화가 지금 변경 범위를 줄이거나 책임 경계를 분명히 하는가
- 구현체가 하나뿐인데도 인터페이스/베이스 클래스가 없으면 곧바로 작업이 막히는가
- 새 추상화 없이도 국소 함수 추출, 작은 타입 추가, 호출부 정리로 해결 가능한가
- 위 질문에서 마지막 항목만 `예`면 새 추상화보다 국소 정리를 우선한다

## 데이터와 경계 규칙

- 기본형 집착, 데이터 뭉치, 전역 데이터, 과도한 가변 데이터는 더 명확한 타입이나 캡슐화로 줄일 수 있는지 검토한다.
- 컬렉션, 레코드성 데이터, 설정 값은 외부에 그대로 노출하지 말고 필요한 책임 위치에서 캡슐화한다.
- 메시지 체인과 과도한 추이적 탐색은 지양하고, 필요한 책임을 더 가까운 타입으로 이동시킬 수 있는지 검토한다.
- 외부 라이브러리나 프레임워크 경계는 깨끗하게 유지하고, 외부 모델이 core 판단 로직을 직접 오염시키지 않게 한다.

## 경계와 데이터 판정 질문

- 현재 변경이 외부 포맷 해석과 비즈니스 판단을 같은 위치에 섞고 있는가
- 항상 같이 전달되는 기본형 묶음이 세 개 이상 반복되는가
- 호출자가 `a.b().c().d()`처럼 여러 단계 내부 구조를 알아야 하는가
- 새 데이터 타입을 추가하지 않아도 경계 책임이 명확한가
- 위 질문 중 둘 이상에 `예`면 캡슐화, 타입 추출, 책임 이동을 우선 검토한다

## 변경 용이성 규칙

- 뒤엉킨 변경, 산탄총 수술, 기능 편애, 거대한 클래스 같은 냄새가 현재 변경으로 더 심해지지 않게 한다.
- 현재 변경이 특정 클래스나 함수에 여러 이유의 수정 압력을 동시에 몰아넣지 않는지 확인한다.
- 설계 개선이 필요하더라도 현재 TASK를 성립시키는 최소 범위를 넘으면 `implementation_notes.md`에 후속 후보로 남긴다.

## 중복과 리팩터링 판단 규칙

- 중복이 보이면 무조건 합치지 말고, 현재 요구사항에서 정말 같은 개념인지 먼저 확인한다.
- 중복 제거가 현재 변경을 단순하게 만들면 반영한다.
- 중복 제거가 새로운 추상화, 새 개념, 더 넓은 영향 범위를 요구하면 이번 TASK에서는 유지하고 후속 후보로 남길 수 있다.
- 리팩터링은 동작 보존을 전제로 하고, 기능 변경과 구조 개선을 동시에 크게 벌리지 않는다.

## 중복 처리 결정 규칙

- 먼저 현재 중복이 정말 같은 개념인지 확인한다.
- 같은 개념이고 현재 변경 범위 안에서 국소 통합으로 줄일 수 있으면 정리한다.
- 같은 모양이지만 다른 책임이거나 앞으로 갈라질 가능성이 크면 유지한다.
- 제거하려면 새 개념, 새 계층, 여러 파일 동시 이동이 필요하면 이번 TASK에서는 유지하고 후속 후보로 남긴다.

## 주석과 문서화 규칙

- 주석은 코드로 표현할 수 없는 의도, 경고, 외부 제약을 남길 때만 사용한다.
- 오래된 주석, 중복 설명, 코드와 어긋난 주석, 주석 처리된 코드는 지양한다.
- 새 구조 결정, trade-off, 범위 밖 후속 작업은 코드 안 임시 주석보다 문서나 `implementation_notes.md`로 남긴다.

## 테스트 가능성 규칙

- 설계는 테스트를 어렵게 만들지 않아야 한다.
- 테스트를 위해 내부 구현을 과도하게 노출하지 않는다.
- 테스트가 지나치게 많은 설정, fixture, mock 협력, 내부 호출 순서를 요구하면 설계가 과도하게 결합됐는지 검토한다.

## design quality 와 performance 경계 규칙

- 시간복잡도, 메모리 사용, 반복 호출, 중간 컬렉션, 성능 검증 계획의 상세 기준은 `docs/process/common/performance_policy.md`를 따른다.
- 병목 근거나 명시된 성능 목표가 없으면 design quality를 크게 해치는 방향의 최적화를 기본 선택으로 삼지 않는다.
- 실제 trade-off가 존재하면 상세 판정 질문과 기록 형식은 `performance_policy.md`를 사용한다.

## 기록 위치 규칙

- 실제 design quality 판단과 trade-off 기록의 1차 위치는 `implementation_notes.md`다.
- `validation_report.md`는 `implementation_notes.md`에 기록된 대안, 추천안, 근거, trade-off의 존재와 요약을 검증한다.
- 코드 주석이나 PR 코멘트만으로 설계 trade-off를 남기지 않는다.

## quality gate 연결 원칙

- design quality는 주로 구현자와 감사자의 수동 판단 영역이다.
- architecture rule, module boundary check처럼 자동화 가능한 일부 항목만 project `docs/project/standards/quality_gate_profile.md`에 연결할 수 있다.
- 이 문서는 특정 정적 분석 도구나 architecture 도구를 강제하지 않는다.

## Phase 2에서 확인해야 할 것

- 현재 변경이 책임 분리, 추상화 수준, 응집도, 경계 분리를 악화시키지 않는가.
- 현재 변경과 직접 관련 없는 대규모 구조 개선이 함께 들어가지 않았는가.
- 중복 처리와 리팩터링 선택이 현재 TASK에서 가장 단순한 방향인가.
- design quality와 performance가 충돌하면 `performance_policy.md` 기준의 대안, 추천안, 근거, trade-off가 식별되는가.

## 빠른 판정 체크리스트

- 함수 분리 판정 질문에 둘 이상 `예`가 나오는데도 책임 이동 없이 유지했는가
- 추상화 도입 판정 질문에서 현재 TASK 이득이 약한데 interface/helper/facade를 추가했는가
- 중복 처리 결정 규칙 없이 "보기 좋아 보여서" 추상화를 택했는가
- 구조 악취를 줄였는지 아닌지 예/아니오로 설명하지 못한 채 broad principle만 반복하고 있는가
- design quality와 performance 충돌이 있었는데 `performance_policy.md` 기록 형식 없이 결론만 남겼는가

## 승인 불가 예시

- 현재 요구보다 앞선 interface/helper/manager 계층 추가
- 하나의 클래스나 함수에 orchestration, domain 규칙, 외부 연동을 함께 몰아넣음
- 플래그 인수와 긴 매개변수 목록으로 분기 책임을 숨김
- 메시지 체인과 경계 누수로 외부 세부사항이 core 판단 로직에 스며듦
- design quality를 명분으로 현재 TASK와 무관한 대규모 재설계나 구조 이동을 함께 반영함

## 승인 가능 예시

- 같은 함수 안의 검증/변환/저장을 국소 함수 둘셋으로 나눠 책임이 더 분명해짐
- 중복처럼 보였지만 다른 책임이라 유지하고, 후속 태스크 후보로만 남김
- interface 추가 대신 작은 함수 추출과 이름 개선으로 현재 요구를 해결함
- 성능 최적화 요구가 있었지만 병목 근거가 없어 더 단순한 구조를 유지하고 trade-off를 기록함
- 외부 DTO 매핑과 domain 판단을 분리하고, 호출자가 메시지 체인을 따라가지 않게 캡슐화를 추가함
