# Testing Policy

## 목적

이 문서는 구현 태스크에서 공통으로 적용되는 테스트 원칙을 정의한다.

## 언제 다시 읽는가

- Phase 2 시작 시 반드시 재확인한다.
- 테스트 범위, 실패 테스트 우선 원칙, 테스트 더블 선택, 통합 테스트 위임 판단이 애매할 때 다시 참조한다.
- Phase 3에서 통합 테스트 대상을 선정할 때 다시 확인한다.

## 기본 원칙

- 테스트는 구현 세부사항보다 동작과 계약을 검증해야 한다.
- 실패하는 테스트를 먼저 작성하고, 테스트를 통과시키는 최소 구현을 반영한다.
- 핵심 규칙과 주요 분기는 명시적으로 검증한다.
- 테스트는 현재 TASK 범위 안에서 필요한 책임을 충분히 덮어야 한다.
- 테스트 더블 선택은 `docs/process/common/test_double_policy.md`를 따른다.

## 단위 테스트 원칙

- 가능한 한 빠르고 결정적인 테스트를 우선한다.
- 외부 시스템, 시간, 랜덤성, 네트워크 같은 변동 요인은 통제 가능한 방식으로 다룬다.
- 분기 제어와 입력 통제는 우선 `stub`로 해결하고, 상태 기반 대체 구현이 필요할 때만 `fake`를 쓴다.
- `mock`는 경계 위임이나 부수효과 호출 의무를 검증할 때만 제한적으로 쓴다.
- 단위 테스트가 부적절한 책임은 억지로 테스트 더블로 덮지 말고 Phase 3로 위임한다.
- 위임한 책임은 `implementation_notes.md`에 기록한다.

## 통합 테스트 원칙

- 레이어 간 연결, 외부 연동 번역, 저장소별 경계 책임, 실제 조립 책임은 통합 테스트에서 검증한다.
- 통합 테스트는 단위 테스트로 대체할 수 없는 책임만 대상으로 삼는다.
- 통합 테스트는 실제 협력 객체를 우선하고, 외부 경계에서만 제한적으로 `stub` 또는 `fake`를 사용한다.
- 불가피한 경우에만 소유한 최외곽 부수효과 경계에서 예외적으로 `mock`를 사용할 수 있다.
- 통합 테스트 안에서 내부 협력 관계를 `mock`으로 대체해 연결 책임을 지우지 않는다.

## 통합 테스트 책임 구분 기준

- 핵심 happy path와 주요 failure path는 통합 수준에서 실제 연결 책임과 함께 검증한다.
- 단위 테스트만으로 확인되지 않는 연결 책임을 통합 테스트 우선 대상으로 삼는다.
- 통합 테스트는 반드시 전체 애플리케이션 end-to-end를 의미하지 않는다.
- 외부 port 구현체, repository, gateway, client adapter처럼 실제 외부 의존성과 연결되는 구현체 단위 검증도 통합 테스트에 포함된다.
- entrypoint, bootstrap처럼 실제 조립과 실행 경계가 핵심인 지점은 단위 테스트보다 통합 테스트가 더 적절할 수 있다.
- adapter는 순수 변환 로직과 실제 연결 책임을 분리해 테스트한다.
- adapter의 순수 변환 로직, 분기, 예외 매핑 규칙은 단위 테스트로 빠르게 검증한다.
- adapter의 실제 외부 호출, 저장소별 경계 책임, 설정 조합, 연결 상태 의존성은 통합 테스트로 검증한다.
- 외부 port 구현체의 실제 저장/조회 연산, driver 또는 client 연동, serialization/deserialization, 저장소별 경계 책임, 응답 또는 레코드 매핑, 에러 번역은 구현체 단위 통합 테스트로 검증할 수 있다.
- 통합 테스트는 아래 책임이 포함될 때 우선 검토한다.
  1. 여러 레이어를 거치며 계약이 유지되는지 확인해야 하는 경우
  2. 외부 요청/응답, 저장 포맷, 에러 번역이 실제 연결에서 맞는지 확인해야 하는 경우
  3. 실제 조립 결과에 따라 동작이 달라지는 entrypoint, bootstrap, adapter 경계인 경우
  4. 단위 테스트 더블로는 실제 실패 양상이나 연결 책임이 가려지는 경우

## 구현체 단위 통합 테스트 예시

- storage adapter 구현체가 실제 저장/조회 연산, 일관성 조건, 조건부 갱신, 레코드 또는 문서 매핑을 기대한 방식으로 수행하는지 검증한다.
- HTTP client adapter 구현체가 실제 응답 포맷을 기대한 DTO 또는 에러로 번역하는지 검증한다.
- 메시지 브로커 publisher/consumer 구현체가 실제 serialization, routing, 실패 처리를 기대한 방식으로 수행하는지 검증한다.
- 이런 테스트는 전체 API를 모두 실행하지 않아도 되며, 해당 구현체와 필요한 실제 의존성만 연결해 수행할 수 있다.

## 프로젝트별 세부 기준

- 실제 테스트 프레임워크, coverage 기준, 통합 테스트 대상, 외부 의존성 기동 방식은 프로젝트 `docs/project/standards/testing_profile.md`에서 정의한다.
- test 실행 명령과 test 게이트의 필수 여부, 실패 처리 기준은 프로젝트 `docs/project/standards/quality_gate_profile.md`에서 정의한다.
