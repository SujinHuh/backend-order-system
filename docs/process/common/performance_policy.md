# Performance Policy

## 목적

이 문서는 Phase 2 구현과 감사에서 시간복잡도, 메모리 사용, 불필요한 중간 컬렉션, 반복 호출, 중복 계산, 과도한 I/O를 같은 기준으로 검토하기 위한 공통 규칙을 정의한다.

## 언제 다시 읽는가

- Phase 2 구현 시작 전에 다시 확인한다.
- 큰 컬렉션 처리, 반복 호출, 캐시, 배치, 스트리밍, 직렬화, 복사 비용이 애매할 때 다시 참조한다.
- 성능 이슈를 주장하거나 성능 때문에 구조를 바꾸려 할 때 다시 읽는다.
- Phase 2 감사 직전에 성능 관점의 확인 항목을 다시 맞출 때 참조한다.

## 기본 원칙

- 성능 품질은 "가장 빠른 코드"가 아니라 현재 요구사항과 입력 규모를 무리 없이 처리하는 합리적 구조다.
- 성능 최적화는 추측보다 근거를 우선한다.
- 현재 TASK와 직접 관련 없는 마이크로 최적화나 광범위한 선제 최적화는 기본 허용하지 않는다.
- 언어별 미세 최적화 테크닉 목록은 이 문서의 범위가 아니다.

## 성능 검토가 필요한 상황

- 현재 변경이 큰 컬렉션, 대량 배치, 큰 파일, 스트림, 페이지네이션 없는 전체 조회를 다룬다.
- 같은 데이터에 대한 반복 순회, 중첩 루프, 반복 정렬, 반복 필터링, 반복 파싱이 들어간다.
- DB, HTTP, 파일 시스템, 메시지 브로커, 캐시 같은 외부 I/O 호출이 루프 안에 있다.
- 큰 객체, 큰 문자열, JSON payload, byte buffer, 이미지, 문서 데이터를 여러 번 복사하거나 전체 메모리에 올린다.
- 명시된 응답 시간 목표, timeout, batch window, throughput, 메모리 제한이 있다.
- 이미 관찰된 timeout, latency 증가, query 폭증, 메모리 압박, OOM, CPU 급증 같은 증상이 있다.

## 성능 검토 트리거 규칙

- 명시된 성능 제약이나 관찰된 증상이 있으면 성능 검토는 필수다.
- 위 상황 중 둘 이상이 동시에 해당하면, 실제 병목이 아직 관찰되지 않았더라도 최소한의 성능 검토를 수행한다.
- 성능 검토를 했으면 결과가 "문제 없음"이어도 근거를 짧게 남긴다.
- 성능 민감 경로인지 애매하면 입력 규모, 호출 빈도, 외부 I/O 존재 여부부터 먼저 확인한다.

## 시간복잡도 규칙

- 현재 변경에서 핵심 입력 단위가 무엇인지 먼저 정한다.
- 입력 증가에 따라 비용이 어떻게 늘어나는지 설명할 수 있어야 한다.
- 같은 데이터에 대한 중복 순회, 중첩 탐색, 반복 정렬, 반복 그룹핑은 정말 필요한지 먼저 확인한다.
- 더 나쁜 시간복잡도를 도입했다면 현재 요구사항과 입력 규모에서 왜 허용 가능한지 설명 가능해야 한다.
- 작은 국소 정리, 인덱싱, 캐싱, 단일 순회화로 줄일 수 있는데 근거 없이 더 비싼 구조를 유지하지 않는다.

## 시간복잡도 판정 질문

- 핵심 입력 크기 `n`이 요청 수, 레코드 수, 파일 크기, 배치 크기처럼 실제로 커질 수 있는가
- 같은 컬렉션이나 데이터 소스에 대해 중첩 루프 또는 반복 탐색이 있는가
- 정렬, 파싱, 직렬화, 정규식, 암호화처럼 상대적으로 비싼 작업을 루프 안에서 반복하는가
- 같은 계산 결과를 같은 요청 또는 배치 안에서 다시 구하고 있는가
- 단일 순회, 사전 인덱스, 배치 조회, 사전 계산으로 비용을 줄일 수 있는가
- 위 질문 중 둘 이상에 `예`면 시간복잡도 검토 결과를 남긴다

## 메모리 사용 규칙

- 전체 데이터를 한 번에 메모리에 올리지 않아도 되면 chunk, stream, iterator, pagination 같은 더 작은 단위를 먼저 검토한다.
- 큰 객체 복사, 큰 문자열 결합, 전체 payload 복제, 직렬화와 역직렬화 반복은 필요 최소 수준으로 줄인다.
- 요청이나 배치가 끝나도 오래 남는 리스트, 맵, 캐시, 버퍼가 생기지 않게 한다.
- 메모리 절감을 위해 구조를 바꾸더라도 현재 요구사항과 설계 품질을 함께 설명할 수 있어야 한다.

## 메모리 판정 질문

- 현재 변경이 전체 결과를 메모리에 materialize하지 않아도 되는가
- 같은 데이터를 새 컬렉션, DTO, 문자열, buffer로 두 번 이상 복사하는가
- 중간 컬렉션이 두 번 이상 연속 생성되는가
- 현재 요청이나 배치 범위를 넘는 캐시나 보관 구조를 새로 추가하는가
- chunk 처리, lazy 처리, 더 작은 표현으로 바꿔도 요구사항을 만족하는가
- 위 질문 중 둘 이상에 `예`면 메모리 검토 결과를 남긴다

## 중간 컬렉션과 큰 객체 복사 규칙

- filter, map, sort, group, flatten, collect 같은 단계가 이어질 때 불필요한 중간 컬렉션이 몇 번 생기는지 확인한다.
- 읽기 쉬움을 위해 단계 분리를 하더라도, 데이터 규모가 큰 경로라면 중간 결과 생성 비용을 함께 판단한다.
- 같은 payload를 domain, transport, logging용으로 반복 복사하지 않는다.
- 큰 객체 복사가 필요하면 왜 참조 공유, 부분 추출, 국소 변환으로 대체할 수 없는지 설명 가능해야 한다.

## 반복 호출, 중복 계산, 과도한 I/O 규칙

- 같은 요청, 배치, 트랜잭션 안에서 동일한 외부 조회나 계산을 반복하지 않는다.
- DB, HTTP, 파일 I/O, 캐시 조회, 템플릿 로딩, 설정 로딩, 스키마 파싱 같은 작업을 루프 안에 두기 전에 배치 가능성을 확인한다.
- 캐시를 도입할 때는 적중 이점뿐 아니라 invalidation, 수명, 일관성 비용도 함께 본다.
- 과도한 I/O를 줄이기 위해 배치나 prefetch를 쓰더라도, 필요 이상으로 넓은 선조회나 메모리 적재를 새 기본값으로 만들지 않는다.

## 반복 호출 판정 질문

- 같은 값이나 같은 조회 결과를 같은 요청 또는 배치 안에서 두 번 이상 다시 구하는가
- 외부 I/O 호출이 아이템별 반복 구조 안에 들어가 있는가
- 같은 파일, 설정, 스키마, 템플릿, 정규식을 반복 생성하거나 반복 읽는가
- 지역 변수 재사용, 국소 캐시, 배치 호출, 조회 합치기로 줄일 수 있는가
- 위 질문 중 둘 이상에 `예`면 반복 호출 또는 중복 계산 검토 결과를 남긴다

## 근거와 검증 계획 규칙

- 성능 이슈나 최적화 필요성은 가능하면 측정값, 운영 증상, 입력 규모, query 수, timeout, 메모리 사용량 같은 근거로 설명한다.
- 정확한 측정이 어려우면 막연한 추측 대신 대상 경로, 예상 성장 축, 우려 이유, 후속 검증 계획을 남긴다.
- `빠를 것 같다`, `느릴 수 있다`, `메모리를 많이 쓸 듯하다` 같은 문장만으로 결론내리지 않는다.
- 성능 개선을 주장하면 무엇이 좋아지는지와 무엇을 어떻게 확인할지까지 함께 남긴다.

## 검증 계획 작성 규칙

- 검증 계획에는 가능한 한 대상 경로, 입력 규모, 실행 방법, 볼 지표를 포함한다.
- 자동 benchmark, profile, query-count check, load test, 수동 재현 시나리오 중 어떤 방식으로 확인할지 적는다.
- 이번 TASK에서 검증을 실제로 실행하지 못하면 그 사유와 잔여 리스크를 남긴다.
- project에 자동화된 성능 게이트가 있으면 실행 명령은 `docs/project/standards/quality_gate_profile.md`를 따른다.

## 설계 품질과 성능 충돌 규칙

- 더 단순하고 읽기 쉬운 구조가 실제 성능 목표를 충족하지 못한다는 근거가 있을 때만 더 복잡한 구조를 채택한다.
- 근거 없는 최적화 때문에 책임 분리, 테스트 가능성, 변경 용이성을 크게 악화시키지 않는다.
- 더 빠른 구조가 필요하더라도 가능한 한 국소 변경으로 해결하고, 구조 전체를 무겁게 바꾸는 선택은 마지막에 검토한다.
- 충돌이 현재 TASK 범위를 넘어서는 구조 결정이면 사용자 승인 또는 후속 태스크 분리가 필요하다.

## 충돌 발생 판정 질문

- 관찰된 병목, timeout, 메모리 압박, query 폭증, SLA 위반, batch window 초과 같은 근거가 있는가
- 더 단순한 설계안을 택하면 현재 요구사항이나 명시된 성능 목표를 충족하지 못하는가
- 더 빠른 구조를 택하면 책임 분리, 테스트 가능성, 읽기 쉬움, 변경 용이성이 실제로 악화되는가
- 더 작은 국소 최적화로는 문제를 해결할 수 없는가
- 첫째 또는 둘째 질문이 `예`이고 셋째 질문도 `예`면 실제 trade-off로 기록한다
- 성능 근거가 없으면 실제 trade-off가 아니라 추측성 최적화로 본다

## 기록 형식

- 대상 경로: `[어느 함수 / 모듈 / 요청 경로 / 배치 경로인지]`
- 입력 규모 또는 제약: `[레코드 수 / 파일 크기 / SLA / timeout / batch window 등]`
- 관찰 증상 또는 우려 근거: `[측정값 / timeout / query 수 / 메모리 사용 / 성장 추정 근거]`
- 검토한 항목: `[시간복잡도 / 메모리 / 중간 컬렉션 / 반복 호출 / I/O]`
- 대안 1: `[프로젝트 결정 필요]`
- 대안 2: `[프로젝트 결정 필요]`
- 추천안: `[프로젝트 결정 필요]`
- 근거: `[왜 이 선택이 현재 요구사항과 입력 규모에 맞는지]`
- 검증 방법 또는 후속 검증 계획: `[benchmark / profile / query count / 수동 재현 시나리오 등]`
- trade-off: `[얻는 것 / 잃는 것]`
- 지금 하지 않는 대안의 이유: `[프로젝트 결정 필요]`

## 기록 위치 규칙

- 실제 성능 검토와 trade-off 기록의 1차 위치는 `implementation_notes.md`다.
- `validation_report.md`는 `implementation_notes.md`에 남은 근거, 검증 방법, 미실행 사유, 잔여 리스크를 검증하고 요약한다.
- 코드 주석이나 PR 코멘트만으로 성능 판단 근거를 남기지 않는다.

## quality gate 연결 원칙

- benchmark, profile, query-count check, load test, performance smoke test 같은 실행 명령은 project `docs/project/standards/quality_gate_profile.md`에서 정의한다.
- 성능 테스트 데이터 규모, fixture, 외부 의존성 기동 방식, 허용 latency 기준 같은 환경 정보는 project `docs/project/standards/testing_profile.md`에서 정의한다.
- 이 문서는 특정 benchmark 도구나 profiler를 강제하지 않는다.
- 자동화된 게이트가 없어도 Phase 2 구현과 감사에서 성능 검토 자체를 생략하지 않는다.

## Phase 2에서 확인해야 할 것

- 현재 변경이 입력 증가에 따라 불필요하게 더 비싼 시간복잡도를 도입하지 않는가.
- 현재 변경이 큰 객체 복사, 중간 컬렉션 생성, 전체 materialization을 불필요하게 늘리지 않는가.
- 현재 변경이 같은 요청이나 배치 안에서 반복 호출, 중복 계산, 과도한 I/O를 새 기본값으로 만들지 않는가.
- 성능 이슈나 최적화 주장이 있다면 근거와 검증 계획이 식별 가능한가.
- 설계 품질과 성능이 실제로 충돌하면 대안, 추천안, 근거, trade-off가 식별되는가.

## 빠른 판정 체크리스트

- 큰 데이터 경로인데 입력 규모와 비용 증가 축을 설명하지 못하는가
- 반복 호출, 중복 계산, 외부 I/O 루프를 두고도 배치나 재사용 검토 흔적이 없는가
- 읽기 쉬움만 보고 중간 컬렉션과 큰 객체 복사 비용을 전혀 보지 않았는가
- 반대로 추측만으로 복잡한 캐시, 조기 최적화, 광범위한 구조 변경을 넣었는가
- 성능 이슈를 말하면서 검증 방법, 미실행 사유, 잔여 리스크를 남기지 않았는가

## 승인 불가 예시

- 데이터 규모가 커질 수 있는 경로에 중첩 탐색과 반복 I/O를 추가하고도 근거나 검토 기록이 없음
- 같은 payload를 여러 단계에서 복사하거나 전체 materialize하면서 이유를 설명하지 못함
- 관찰된 병목이 없는데 가독성과 책임 분리를 크게 해치는 선제 캐시나 구조 변경을 도입함
- 성능 최적화를 주장하지만 어떤 경로를 어떤 방식으로 확인했는지 또는 확인할지 남기지 않음

## 승인 가능 예시

- 큰 컬렉션 경로에서 중간 컬렉션 두 번을 단일 순회로 줄이고, 근거와 검증 계획을 함께 남김
- 반복 DB 조회를 배치 조회로 바꾸되 메모리 적재 범위가 커지지 않도록 현재 페이지 범위로만 제한함
- 실제 병목 근거가 없어 더 단순한 구조를 유지하고, 왜 최적화를 미루는지 `implementation_notes.md`에 남김
- 더 빠른 구조가 필요했지만 함수 분리와 이름 정리로 책임 분리를 최대한 유지하며 국소 변경으로 해결함
