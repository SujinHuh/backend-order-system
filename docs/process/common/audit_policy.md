# Audit Policy

## 목적

이 문서는 모든 Phase 감사에서 공통으로 적용되는 운영 규칙을 정의한다.

## 언제 다시 읽는가

- 각 Phase에서 감사를 수행하기 직전에 다시 읽는다.
- 같은 Phase를 재감사하기 직전에는 이전 피드백 해소 여부 확인 규칙을 다시 읽는다.
- 감사 결과 형식, 피드백 분류, 미확인 입력물 표기 방식이 애매할 때 다시 참조한다.

## 공통 감사 입력물 확인 규칙

감사자는 Phase 전용 입력물 외에도 아래 항목을 함께 확인한다.

- 해당 task의 `issue.md`
- 해당 task의 `phase_status.md`
- 해당 task의 `implementation_notes.md`
- 해당 task의 현재 Phase 산출물
- 이전 Phase에서 현재 Phase 판단에 영향을 주는 산출물
- 관련 공통 정책 문서
- 관련 프로젝트 전용 문서

입력물 중 일부를 읽지 못했거나 존재하지 않으면 그 사실을 감사 결과에 명시한다.

- 경량 운영 예외가 적용됐거나 후보였으면 `docs/process/common/lightweight_task_policy.md`도 함께 확인한다.
- 현재 작업이 runtime contract, phase gate, entrypoint chain, downstream 사용 흐름을 바꿨다면 아래 인접 문서도 함께 교차 검토한다.
  - `docs/entrypoint.md`
  - `docs/process/harness_guide.md`
  - `docs/project/standards/architecture.md`
  - `docs/project/standards/implementation_order.md`

## 공통 감사 운영 규칙

- 감사는 해당 Phase 산출물과 관련 입력물을 기준으로 수행한다.
- 모든 감사와 재감사는 구현 주체와 분리된 서브에이전트가 수행한다.
- 감사자는 최신 산출물 기준으로만 판정하며 stale 상태의 이전 감사 결과나 사용자 승인을 승인 근거로 재사용하지 않는다.
- 감사가 승인 불가이면 피드백을 반영한 뒤 같은 Phase에서 재감사한다.
- 재감사는 이전 감사 피드백이 해소되었는지 먼저 확인한 뒤 남은 문제와 신규 문제를 구분해 기록한다.
- 감사자는 범위 위반, 책임 혼합, 테스트 부족, 검증 부족, 문서 반영 누락, 사용자 승인 누락을 우선적으로 본다.
- self-healing 이 발생한 task라면 감사자는 가장 이른 영향 Phase 선언, stale 처리 범위, write-set lock, 잠긴 산출물 위반 여부가 `implementation_notes.md`와 실제 변경 파일에 일치하는지 먼저 확인한다.
- `phase_status.md`가 있는 task라면 감사자는 현재 gate, 허용 write-set, 잠긴 경로, stale 산출물이 실제 변경 파일과 일치하는지 확인하고 필요하면 `scripts/validate_phase_gate.py` 결과를 함께 본다.
- 감사자는 중요한 알고리즘 선택, 구조 변경, 복잡도 trade-off 결정이 있으면 사용자 승인 여부와 기록 위치를 확인한다.
- 감사자는 프로젝트 전용 규칙이 필요한 판단이면 overlay 문서를 함께 참조한다.
- 감사자는 현재 판단이 repo-local 근거를 우선했고, 기억/외부 대화/다른 프로젝트 관행으로 repo 빈칸을 추측 채우지 않았는지 확인한다.
- 감사자는 현재 변경이 공통 code quality 기준과 프로젝트 convention을 약화하지 않는지, 유지보수성을 떨어뜨리는 새 패턴이나 drift를 만들지 않는지 확인한다.
- 감사자는 승인된 작업공간과 문서 집합 밖의 변경이 섞이지 않았는지 먼저 확인한다.
- 경량 운영 예외가 적용된 task라면 경량 승인 상태가 최신 입력 기준으로 여전히 유효한지, Full 전환 트리거가 있었는데도 경량 예외를 유지하지 않았는지 확인한다.
- 감사 결과가 승인 가능이어도 다음 Phase 진행 여부는 사용자 승인 게이트를 통과해야 확정된다.

## 공통 감사 피드백 분류 규칙

피드백은 가능하면 아래 분류 중 하나 이상으로 구분한다.

- 누락
- 모호함
- 구조 위반
- 범위 확장
- 승인 누락
- stale 재사용
- 테스트 부족
- 검증 부족
- 문서화 부족

## 감사 결과 형식

- 판정: 승인 가능 / 승인 불가
- 재감사인 경우 이전 피드백 해소 여부: 항목별 확인 결과
- 사용자 승인 확인: 필요한 승인 항목과 확인 결과
- stale 상태 확인: 없으면 없음, 있으면 stale 처리된 감사/승인/산출물과 재적용 필요 게이트
- 피드백: 보완이 필요한 항목 목록
- 피드백 분류: 위 분류 기준 사용
- 미확인 입력물: 없으면 없음으로 표기
