# Test Double Policy

## 목적

이 문서는 공통 Harness에서 `mock`, `stub`, `fake`를 언제 사용하는지 구분하는 core 수준 원칙을 정의한다.

## 언제 다시 읽는가

- Phase 2에서 단위 테스트용 테스트 더블을 고를 때 다시 읽는다.
- Phase 3에서 통합 테스트에서 실구현 대신 테스트 더블을 써도 되는지 판단할 때 다시 읽는다.
- 테스트 더블이 현재 검증 책임을 흐리거나 구현 세부사항에 과도하게 결합되는지 애매할 때 다시 참조한다.

## 기본 원칙

- 테스트 더블은 구현 구조를 흉내 내기 위한 도구가 아니라 현재 검증 책임을 분명하게 만들기 위한 도구여야 한다.
- 현재 책임을 검증하는 데 더 단순한 테스트 더블로 충분하면 더 강한 테스트 더블을 쓰지 않는다.
- 테스트 더블이 많아질수록 현재 테스트가 구현 구조에 과하게 결합되었는지 의심한다.
- 테스트 더블이 없으면 현재 책임을 설명할 수 없을 정도로 경계가 흐리면 설계를 다시 보거나 적절한 Phase로 위임한다.
- 특정 언어나 프레임워크의 mocking 문법, spy API, patch 문법은 core가 아니라 project overlay에서 정의한다.

## 테스트 더블 구분

### `stub`

- 입력에 대해 미리 정한 값이나 에러를 돌려주는 더블이다.
- 호출 여부나 호출 순서를 검증하지 않는다.
- 분기 제어, 실패 경로 재현, 시간/랜덤/네트워크 같은 변동 요인 통제에 우선 사용한다.

### `fake`

- 실제 구현 대신 동작하는 단순 대체 구현이다.
- 메모리 기반 저장소나 경량 대체 서비스처럼 의미 있는 계약을 유지하면서도 실행 비용을 낮출 때 사용한다.
- 실제 시스템의 핵심 계약을 유지할 수 있을 때만 사용하고, 의미 있는 차이가 있으면 그 차이를 기록한다.

### `mock`

- 특정 상호작용이나 위임, 부수효과 호출 의무를 검증하는 더블이다.
- 결과값만으로는 책임을 충분히 검증할 수 없고, `어떤 경계 호출을 반드시 수행해야 하는가`가 핵심일 때 사용한다.
- 구현 내부 협력 순서나 세부 호출 횟수를 과도하게 고정하는 용도로 쓰지 않는다.

## 단위 테스트에서의 선택 원칙

- 단위 테스트는 기본적으로 `stub`를 우선한다.
- 외부 조회 결과, 시간, 랜덤성, 네트워크 응답, 에러 분기를 제어할 때는 `stub`를 사용한다.
- 상태 변화나 저장 결과 같은 의미 있는 계약을 검증해야 하고 경량 대체 구현으로 충분하면 `fake`를 사용한다.
- 현재 유스케이스의 핵심 책임이 특정 경계로의 위임 또는 부수효과 호출 의무라면 `mock`를 사용한다.
- 같은 레이어 내부 협력의 세부 호출 순서, 중간 호출 횟수, private한 조립 과정을 `mock`로 고정하지 않는다.
- 경계 번역, 실제 조립, 외부 계약 적합성처럼 단위 테스트 더블로 덮으면 책임이 흐려지는 경우는 Phase 3 통합 테스트로 위임한다.

## 통합 테스트에서의 선택 원칙

- 통합 테스트는 가능한 한 실제 협력 객체와 실제 연결을 우선 사용한다.
- 통합 테스트 안에서 내부 협력 관계를 `mock`으로 대체해 연결 책임을 지우지 않는다.
- 실제 외부 시스템을 쓰기 어렵거나 비용이 큰 경우에만 외부 경계에서 제한적으로 `stub` 또는 `fake`를 사용한다.
- 외부 시스템의 특정 에러, timeout, 재시도 분기처럼 재현이 어려운 실패 조건은 `stub`로 제어할 수 있다.
- 실제 시스템의 핵심 계약을 유지하는 경량 대체 구현이 있을 때만 `fake`를 사용한다.
- 통합 테스트에서 `mock`가 필요하다면 소유한 최외곽 부수효과 경계에 한정하고, 왜 실제 연결 대신 상호작용 검증이 필요한지 설명할 수 있어야 한다.

## 선택 실패 신호

- 테스트가 내부 구현 순서나 호출 구조를 과도하게 알아야만 통과한다.
- `mock`를 여러 개 겹쳐야만 테스트를 작성할 수 있다.
- `fake`가 실제 계약과 다르게 동작하는데도 그 차이를 무시하고 있다.
- `stub`로 외부 계약 적합성이나 경계 번역 책임까지 검증하려고 한다.
- 통합 테스트가 실제 연결 책임 대신 더블 설정만 검증하고 있다.

위 신호가 보이면 테스트 더블 선택을 다시 보고, 필요하면 설계 조정 또는 Phase 3 위임 여부를 검토한다.
